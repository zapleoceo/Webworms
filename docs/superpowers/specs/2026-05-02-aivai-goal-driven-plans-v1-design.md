# AIVAI: Goal‑Driven Plans v1 (Reachable Move + Rope Feasibility + Grenade Retreat) + Grenade UI

## Контекст

Текущее поведение AIVAI показывает системные проблемы:

- Move: бот “упирается” в стены/ямы и повторяет неэффективные попытки вместо смены цели/маршрута.
- Rope: контроллер часто получает `dx_small` и либо спамит попытки, либо банит стратегию; воркер не проверяет feasibility rope до выдачи цели.
- Grenade: выбор гранаты делается как “одиночный выстрел”, без модели “бросок + отход”, поэтому возникают “положил под ноги” и self/FF; при этом граната дефицитна (15 на команду).
- UI: игроку нужно видеть остаток гранат по командам во время матча.

Параллельно важны ограничения по ресурсам: планирование выполняется в worker и должно быть предсказуемым по CPU/GC.

## Цели

1) Move‑план всегда “достижимый”: воркер не должен выдавать `moveTo`, если нет пути, и должен отдавать `movePath` (waypoints).
2) Rope учитывается как действие только если оно реально осуществимо в этой геометрии (anchor/углы/дистанции), без спама.
3) Grenade превращается в 2‑шаговый план: `throw` + `retreat` в safe‑позицию, и граната запрещается, если отход недостижим.
4) UI показывает остаток гранат по командам: иконка гранаты + число (0..15) под карточками в углах; карточки уменьшить на ~20%.

## Не‑цели (v1)

- Полная модель “rope + swing speed + jump release” (сложно, высокий риск). В v1 только feasibility attach и базовые режимы (swing/climb/descend) как сейчас.
- Полное тактическое планирование на несколько ходов вперёд.
- Идеальная оптимизация “бросок гранаты с учётом всех вариантов покатиться/рикошета”. Используем существующую симуляцию и практичные эвристики.

## Основная идея: план как последовательность

Вместо “moveTo + action” план становится последовательностью стадий:

- `movePath` (waypoints + primitive) — как сейчас
- `action` (выстрел/копка/др.) — как сейчас
- `postMovePath?` — новый опциональный шаг (в v1 нужен для grenade retreat)

Ключевое правило: план может не содержать `action` (если нет разумного выстрела), и может содержать `movePath` без выстрела.

## v1: Reachability‑gated Move

### Воркера

Перед тем как вернуть `plan.moveTo`, воркер обязан проверить путь:

- `path = findWaypointPath(terrain, shooterPos, moveTo, ...)`
- если пути нет:
  - выбрать другой кандидат `moveTo` из ограниченного списка (см. “кандидаты позиций” ниже)
  - либо вернуть `plan` без `moveTo` (только fire, если текущая позиция ок)
- если путь есть:
  - вернуть `plan.movePath = { waypoints, primitive }`
  - `plan.moveTo` можно оставить как финальную точку для UI/логов, но main должен следовать `movePath`.

### Кандидаты позиций (минимально)

В v1 не делаем огромный поиск. Берём:

- позицию “на линии огня” как сейчас (worker выбирает firing‑позицию)
- 6–10 точек вокруг текущей позиции (±X, ±Y по поверхности) с быстрым фильтром “на земле”
- 2–4 точки ближе к цели/врагу (эвристика)

Каждый кандидат проходит reachability check через `findWaypointPath`.

### На main (контроллер движения)

При отсутствии прогресса:

- если `dx/dy` к текущему waypoint не уменьшается за N секунд и `stuckTime` растёт → сбрасывать `movePath` и просить replanning (у вас частично есть).

Ключевой рефактор: replanning должен менять цель (через воркер), а не повторять тот же `moveTo`.

## v1: Rope Feasibility (без спама)

### Проблема

Сейчас воркер может вернуть цель, где “по идее” нужен rope, но main‑контроллер на месте спамит `rope_swing` и получает `dx_small`/`no_anchor`.

### Решение v1

1) В воркере: при построении графа пути (или в отдельной проверке) определить, есть ли rope‑вариант:
   - использовать уже существующую проверку якоря в PathPlanner (упрощённо) как сигнал feasibility.
   - если `allowRope=false` (ropeRemaining=0) — не использовать.

2) В main: “soft failures” не должны банить стратегию:
   - `cooldown` и `dx_small` — это не причина “запретить rope навсегда”, это сигнал “попробуй другое действие или дождись окна”.

3) Убрать жёсткие магические отсечки для `rope_swing`:
   - в v1 либо уменьшить/зависимо от режима ослабить порог,
   - либо принимать “rope только если воркер сказал primitive=rope” (чтобы main не изобретал rope, когда он не в плане).

## v1: Grenade Throw + Retreat

### Проблема

“Положил под ноги” и self/FF возникают потому что:

- grenade выбирается как одиночный выстрел;
- safety оценивается в текущей позиции;
- нет обязательного отхода.

### Решение v1

Добавить для grenade условие применимости:

1) Воркер выбирает grenade‑кандидата (как сейчас) и получает `impact` (точку детонации/первого контакта).
2) Строит `retreatCandidates` вокруг стрелка:
   - точки в радиусе 120–220px (на земле), предпочтительно “в сторону от impact”.
3) Для каждой точки ищет путь `findWaypointPath(shooterPos -> retreatPos)` и проверяет, что:
   - путь существует,
   - путь короткий по расстоянию (proxy времени),
   - retreatPos находится за `safeRadius` от impact.
4) Если safe retreat найден:
   - вернуть `action = grenade`
   - вернуть `postMovePath = retreatPath`
5) Если safe retreat не найден:
   - grenade запрещается (кандидат отбрасывается) независимо от expectedDamage.

### Ресурсная цена гранаты

Граната дефицитна:

- если `teamAmmo[team].grenade` мал, вводить штраф (уже частично сделано)
- дополнительно: grenade запрещается в `reserve_fire` без safe retreat

## UI: Grenade Counter

### Требования

- В левом и правом верхнем углу (рядом с карточками команд):
  - карточки уменьшить на ~20%
  - под карточкой показывать иконку гранаты + число оставшихся гранат у команды
- Значение берётся из `presenter.state.teamAmmo.teamX.grenade`
- Обновление: при каждом изменении (лучше в render/update loop презентара)

### Минимальная реализация

- Добавить 2 контейнера `team1`/`team2` рядом с существующими HUD карточками.
- Рендерить: `<img ...grenadeIcon...> <span>15</span>`
- Если гранат `Infinity`/undefined — скрывать (на случай режимов без лимита).

## Логи и диагностика (чтобы быстрее тюнить)

Добавить/расширить события:

- `bot_decision` уже содержит `weaponId` (это хорошо).
- Добавить в `bot_decision.debug`:
  - `movePathLen`, `postMovePathLen`
  - `retreatUsed` (0/1)
  - `grenLeft`

## План внедрения (итерациями)

1) Worker: reachability‑gated moveTo + всегда отдавать movePath (если moveTo выбран).
2) Grenade retreat: postMovePath + запрет grenade без safe retreat.
3) Main: исполнение postMovePath после выстрела (grenade).
4) UI: счётчик гранат + уменьшение карточек.
5) Тюнинг rope feasibility (встроить в воркер и не спамить в main).
