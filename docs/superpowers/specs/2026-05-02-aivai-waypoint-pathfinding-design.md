# AIVAI: Waypoint Pathfinding + Match ID UI

## Контекст и проблема

В режиме `mode=aivai` боты часто:

- простаивают значительную часть хода, ожидая план от воркера;
- упираются в стену, продолжая пытаться идти вперёд;
- редко используют прыжок/верёвку для обхода препятствия;
- стреляют в конце хода по резервной логике и часто нерезультативно.

Цель: заменить эвристику движения “walk/jump/rope по сенсорам” на планирование маршрута по waypoint’ам (грид + A*) и дополнительно вывести `matchId` на экран окончания матча AIVAI.

## Цели

- Бот на каждом ходу имеет маршрут из нескольких waypoint’ов и явно выбранный примитив движения для перехода к следующей точке.
- Исполнитель движения следует плану и делает replanning при отсутствии прогресса.
- Логирование остаётся совместимым; добавляется диагностическая информация о пути (минимально).
- На экране конца матча (AIVAI) отображается `matchId` + удобное копирование.

## Не-цели

- Не делаем полноценный “воксельный” физический планировщик.
- Не пытаемся гарантировать 100% результативность каждого выстрела на любой карте; улучшаем навигацию и устойчивость принятия решений.
- Не меняем PVP/онлайн режимы; изменения касаются `ai`/`aivai` исполнителя.

## Определения

- **Waypoint**: позиция `{x,y}` в пикселях (центр червя).
- **Node**: дискретизированный waypoint на сетке.
- **Move primitive**: `walk | jump | rope`.
- **Path plan**: список waypoint’ов + следующий примитив движения.

## Высокоуровневая архитектура

### Компоненты

1) `BotThinkWorker` (уже существует)
- получает snapshot мира и параметры хода;
- возвращает `planResult` (уже существует).

2) Новый `PathPlanner` внутри воркера
- строит навигационный граф вокруг текущей позиции;
- ищет маршрут к “goal region” (см. ниже);
- возвращает `path` (waypoints) + `nextPrimitive`.

3) `BotTurnController` (на главном потоке)
- оркестратор: тайминг, выбор между worker планом и резервом;
- исполнитель: выполняет `nextPrimitive` для достижения текущего waypoint;
- replanning: триггер при stuck / отсутствии прогресса.

4) UI (клиент)
- отображает `AIVAI matchId` на экране окончания матча.

## Данные и контракт воркер ↔ основной поток

### Новый формат результата воркера

`planResult.plan` расширяется:

- `movePath?: {`
  - `waypoints: Array<{x:number,y:number}>`
  - `primitive: 'walk'|'jump'|'rope'`
  - `primitiveParams?: {`
    - для `jump`: `{ dir: 'left'|'right', holdMs?: number }` (минимально)
    - для `rope`: `{ anchorHint?: {x:number,y:number}, mode?: 'climb'|'swing'|'descend' }` (минимально)
  - `}`

Существующие поля (`moveTo`, `action`, `intent`, `intentReason`) остаются для совместимости на время миграции.

## Сетка навигации и валидность узлов

### Параметры сетки

- Grid step: 16px.
- Окно планирования: прямоугольник вокруг shooter и ближайшей цели:
  - `x`: `[min(shooter.x, target.x)-R, max(...)+R]`, `R=480`
  - `y`: `[0..h]`, но фактически ограничиваем интересный диапазон `shooter.y±320`.

### Валидность Node (позиция червя)

Узел `{x,y}` валиден, если:

- тело не пересекает террейн: несколько точек внутри AABB червя имеют material=0;
- есть опора: под “ступнями” в пределах 2–6px есть material>0;
- позиция в пределах мира.

Ожидается утилита в воркере: `probeMaterial(x,y)` на основе ландшафт snapshot’а.

## Рёбра (Move primitives)

### Walk edges

- Соседние `x±step` с корректировкой `y` на ближайшую “поверхность” (простая привязка к земле).
- Валидны если:
  - не проходим сквозь стену (проверка линии/нескольких точек между старым и новым x);
  - не шагаем в “deep void” (ограничение максимального падения на шаг).

### Jump edges

- Кандидаты: `x±(2..6)*step`, `y-(1..4)*step` + варианты “плоского прыжка”.
- Валидность: дискретная проверка траектории (8–12 сэмплов) на коллизии с террейном.
- Стоимость выше walk.

### Rope edges (упрощённо)

Rope трактуем как быстрое перемещение к точке с другим `y` или через gap.

- Генерация anchor candidates:
  - набор точек по окружности вокруг игрока (глобальные углы, как уже делается в runtime), но в воркере;
  - anchor валиден если material>0 и “видимость” (линия к anchor без террейна).
- Применение:
  - если есть валидный anchor и достигнутый endpoint близок к нужному node (или уменьшает heuristic) → добавляем rope edge.

Важно: для первой итерации rope edges не обязаны быть идеальными, задача — перестать упираться в стену.

## Goal region: куда строим маршрут

Маршрут строится не “прямо в врага”, а к позиции, где:

- есть хотя бы один приемлемый выстрел по врагу (expectedDamage выше порога);
- self-risk ниже порога.

Воркер для набора кандидат-позиций (например каждые 32px по сетке вокруг) вычисляет:

- доступность выстрела по нескольким дискретным углам/пауэрам (reuse текущей оценочной логики из `BotAI`/MCTS при наличии);
- минимальный self-risk.

Если ни одной “firing position” нет, цель упрощается до “сближение безопасно” (min distance to target без cliff).

## Исполнение плана в BotTurnController

### Основной цикл хода

- Если `movePath.waypoints` есть:
  - активный waypoint = `waypoints[0]` (или индекс).
  - `primitive` задаёт выбор поведения (walk/jump/rope) вместо эвристики `selectStrategy`.
  - при достижении waypoint (dx/dy пороги) → переходим к следующему.

- Replanning triggers:
  - `stuckTime > 0.6` или `cost` не улучшается за `T=0.8s`;
  - `dirFlipCount` аномально высок;
  - waypoint недостижим (порог попыток).
  - Тогда отправляем воркеру новый запрос с banlist (например “rope временно запрещён”).

### Временные ограничения

- Воркер продолжает считаться параллельно, но основной поток должен всегда иметь безопасный fallback (движение к врагу + не стрелять в самоурон).
- `reserve_fire` остаётся, но должен использовать тот же фильтр self-risk.

## UI: matchId на экране окончания матча (AIVAI only)

- В режиме `mode=aivai` на экране `VICTORY!`/окончания матча показываем строку:
  - `AIVAI Match ID: aivai_...`
- Кнопки:
  - `COPY` (копирует `matchId`)
  - опционально `OPEN STATS` (открывает `/api/aivai/log/stats?matchId=...`)

Источник `matchId`: значение, которое уже формируется на клиенте и хранится как `window.__aivaiMatchId`.

## Логи и диагностика

Минимально добавить в AIVAI trace (опционально):

- `bot_path` событие 1 раз за ход:
  - `waypointsCount`, `primitive`, `goalType`, `astarNodesExpanded`

Это нужно чтобы видеть: воркер реально строит путь, а не падает в fallback.

## Тестирование

- Unit tests (воркер/планировщик):
  - строит маршрут вокруг простой стены (walk невозможен → jump или rope появляется);
  - не предлагает узлы внутри террейна;
  - goal region выбирается при наличии shot candidates.

- Integration tests (логика контроллера):
  - при `stuckTime` триггерится replanning;
  - при наличии `movePath` контроллер не использует `selectStrategy` эвристику.

## Риски и смягчения

- Стоимость вычислений: ограничить окно планирования и число узлов.
- Rope edges сложно моделировать: первая версия допускает грубую модель и переносит точную механику на исполнитель.
- Регрессии поведения: флаг/включение только для `aivai` на первом этапе.

1) Добавить структуры `movePath` и протянуть их через воркер → контроллер.

3) Добавить `rope` edges (упрощённо).
