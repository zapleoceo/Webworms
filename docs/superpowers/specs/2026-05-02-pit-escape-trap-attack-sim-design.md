# Pit Escape + Trap Attack + Симулятор (AIVAI)

## Контекст

В `aivai_1777756452486_hard_hard` наблюдаются:

- зацикливание поведения в узких ямах/полостях: юниты продолжают `walk` в стенки вместо целенаправленного выхода
- атакующая сторона недостаточно использует “trap target” ситуацию (граната в яму и т.п.)
- возможны фризы из-за повторяющихся попыток/перепланирования без прогресса

Цель — добавить корректную обработку полостей/ям/щелей разных размеров и добиться стабильных метрик на разных картах.

## KPI

“Ход” = любой ход бота (включая escape/reposition/attack).

- `damageRate >= 0.50`: доля ходов, в которых итоговый `enemyDelta > 0` (после завершения эффекта выстрела/в конце окна оценки)
- `goalRate >= 0.80`: доля ходов, в которых выполнен критерий успеха intent’а (см. ниже)

Оценка производится на батче сценариев из разных карт/seed’ов + синтетических pits.

## Ограничения

- Изменения не должны быть “тяжёлыми на клиенте”: анализ ям должен быть локальным и кэшируемым, без глобальных flood-fill на каждом кадре.
- Допускаются концептуальные изменения стратегии.
- Итерации: до 1000 циклов “изменение → прогон симулятора → лог результата → commit”.

## Термины

- `pit`: локальная полость/яма/щель, куда юнит может провалиться.
- `trapped`: состояние, где простым `walk` (и часто прыжком) нельзя выйти за разумное время из-за стенок/низкого потолка/узкого профиля.
- `trap target`: цель в `trapped`/pit-состоянии, по которой выгодно кидать гранату.

## Intent’ы и критерии успеха

### `attack`

Успех: `enemyDelta > 0` в окне оценки (с существующим `shot_eval` или аналогом).

### `escape_pit`

Успех (любое):

- `isTrapped` стало `false`, или
- `depthPx` уменьшился минимум на `minDepthGainPx`, или
- юнит перешёл на `rimY`/вышел на поверхность (высота/геометрический критерий).

### `reposition_safety`

Успех: увеличили `selfNear`/safety margin (в логике скоринга) минимум на порог или уменьшили edge-risk.

## PitAnalyzer (v1)

Новый модуль, работающий от `Landscape.getMaterial(x,y)` (или `TerrainQuery.isSolid`) и параметров юнита.

Выход:

- `isTrapped: boolean`
- `depthPx: number`
- `widthPx: number` (примерная ширина “колодца” на уровне плеч/головы)
- `rimY: number`
- `rimLeftX: number | null`, `rimRightX: number | null`
- `escapeDir: 'left' | 'right' | 'both' | 'none'`
- `canJumpOut: boolean` (по доступной высоте и свободе над головой)
- `canRopeOut: boolean` (по наличию потенциального якоря выше rimY, без зоны border)
- `thinWallDir: 'left' | 'right' | 'none'` (направление “тонкой стенки”, пригодной для dig)
- `attackVuln: number` (0..1), растёт с глубиной и узостью

Алгоритм (локальный, без глобальных проходов):

- измерение “стенок” слева/справа на уровнях `foot/mid/head` в радиусе `scanX`
- оценка “rim” через поиск ближайшей surface по `scanX` и “глубины” как `rimY - currentY`
- `thinWallDir`: локальный замер “толщины” стены по горизонтали на `mid/head` в пределах `thinScanMax`, выбираем минимум

Кэширование:

- per-turn cache: `key = ${terrainRevision}:${dfEventsCount}:${wormId}:${xBin}:${yBin}`
- ограничение частоты пересчёта (например, не чаще, чем раз в 0.2s, если не менялся `terrainSig`)

## Escape-стратегия (Jump → Rope → Dig)

В `BotTurnController` вводится ранний режим:

1) Если `PitAnalyzer.isTrapped`:
   - intent = `escape_pit`
   - блокируем бессмысленный `walk` в стенку: приоритет “прогресса вверх/к rim”.
2) `Jump`:
   - выбираем `escapeDir` и “подпрыгиваем” с небольшим backoff (микро-позиционирование у стенки)
   - критерий прогресса: уменьшение `depthPx` / достижение `rimY`.
   - ограничение: `KJumpAttempts` до перехода к rope.
3) `Rope`:
   - специальный подбор якоря “над rimY”, а не текущий `moveTo`-ориентированный поиск.
   - избегаем `border` и накопленные `ropeBorder/ropeNoAttach` влияют на выбор.
   - ограничение `KRopeAttempts` до перехода к dig.
4) `Dig`:
   - используем существующую механику, но направление выбирается по `thinWallDir`.
   - если `thinWallDir` = none, используем оба направления по очереди (round-robin), чтобы “в любом направлении, лишь бы выбраться”.

## Trap Attack (граната в яму)

В `chooseBotActionScored` добавляется “trap-bonus”, если цель trapped:

- `score += trapBonus * attackVuln`
- ослабление/обход фильтра `grenade_close_vertical` для `pit target`
- снижение порога `grenadeMinExpectedDamage` для `pit target` (или замена на условие `expectedDamage >= min || trapBonus >= X`)
- расширение candidate space для гранаты в `pit-mode` (высокие углы/средние power), с дедупом по биннингу.

## Симулятор/харнесс

Добавить отдельный “быстрый симулятор” для оффлайн прогона:

- источники terrain:
  - разные `mapType`: `islands/cave/flat`
  - несколько seed’ов на mapType
  - синтетические pits (профильные формы): узкий колодец, широкий бассейн, щель, overhang
- сценарии:
  - trapped-юнит в яме vs свободный противник
  - оба trapped
  - trapped + ограниченный grenade ammo
- метрики:
  - `damageRate`, `goalRate`, а также раздельно `attackGoalRate/escapeGoalRate`

Вывод:

- один markdown/ndjson лог файл в репо: `docs/sim/aivai-pit-iterations.md` (добавляем запись на каждый цикл)
- на каждом цикле: изменение (кратко), метрики по группам, общий KPI

## Итерационный цикл (до 1000)

Каждая итерация:

1) небольшое изменение эвристик/логики (или refactor) в рамках PitAnalyzer/escape/attack
2) `npm test` (или таргетный тест для симулятора)
3) прогон симулятора (батч сценариев)
4) запись результата в лог
5) commit с коротким сообщением

Останов:

- KPI достигнут, или
- 1000 итераций исчерпано
