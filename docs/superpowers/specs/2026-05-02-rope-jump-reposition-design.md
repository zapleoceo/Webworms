## Контекст и симптомы

По AIVAI логам последних матчей:

- `aivai_1777734550301_hard_hard`: 0 выстрелов, 1066 событий `bot_rope_attempt`, из них подавляющее большинство `result:"cooldown"`. Бот тратит ход на повторные попытки верёвки, но прогресса почти нет.
- `aivai_1777734373475_hard_hard`: много `shot_eval.noRes=1` при кажущейся “близости” целей; в debug высокий `muzzle_blocked`, а `bestByWeaponId[*].expectedDamage≈0`. Бот стреляет из позиции, где линия огня фактически перекрыта террейном.
- `aivai_1777733926875_hard_hard`: цепочки `bot_rope_attempt` с `cooldown/dx_small`, стратегия “хочу наверх” не реализуется.

## Цели

1) Уменьшить “тупняк на верёвке”: не спамить попытками в cooldown-окно, повышать шанс реального `attached`.
2) Добавить “jump + rope” (цепляться в полёте) для выхода из ям/колодцев и вертикальных ловушек.
3) Убрать бессмысленные выстрелы (noRes) когда лучшая стрельба имеет `expectedDamage≈0` и высокий `muzzle_blocked`; вместо этого бот должен выбирать reposition (move/rope).
4) Сделать логи верёвки объясняющими причину неуспеха (почему не зацепился).

## Не-цели

- Не вводить жёсткий тайм-лимит на воркер.
- Не переписывать физику RopeTool.

## Изменения (патч “A”)

### A1) Rope попытки: анти-спам и нормальная телеметрия

Текущее:
- `tryAttachRope` вызывается каждый тик, и при `now - lastRopeAttemptAt < 0.55` логируется `cooldown`, без изменения поведения.

Предлагается:
- Внутри rope-стратегии при `cooldown` не “пере-пытаться”, а удерживать релевантные inputs (движение/качание/карабканье) или просто ждать окна.
- Логировать `cooldown` не чаще 1 раза на окно (например, отдельный `lastRopeCooldownLoggedAt`).
- В `bot_rope_attempt` добавить поля:
  - `aimAngle`, `facingRight`
  - `rayHit` (x,y,dist,hit) для луча верёвки
  - `ropeActiveAfterFire` (0/1)
  - `ropeCast` (x,y) из RopeTool (куда реально “кинули”)

### A2) Rope поиск якоря: расширение углов и вертикальные исключения

Текущее:
- Набор углов ограниченный; `dx_small` режет попытки даже когда цель значительно выше.

Предлагается:
- Для climb/swing увеличить дискретизацию углов (добавить ещё 4–8 углов ближе к вертикали и ближе к горизонтали).
- Для “вертикального побега” (dy сильно отрицательный) ослабить `dx_small`:
  - если `dy <= -120` разрешать `dx` меньше текущего порога (или разрешать `bothSides` даже при малом dx).

### A3) Jump + Rope: “rope_jump_attach”

Триггер:
- Цель выше (dy <= -120) и обычный `rope_climb/rope_swing` не дал `attached` за M попыток / или постоянно упирается в `dx_small/no_anchor`.

Поведение:
- Бот делает `jump` в сторону цели (dir), затем в течение короткого окна полёта (например пока `vy < 0` или первые 0.25–0.35с) пробует `tryAttachRope` с расширенными углами.
- Если `ropeActive` включился, переходит в обычный `executeRope`.
- Если окно прошло и `ropeActive` нет, стратегия считается неуспешной и бот делает replan/bounce в другой вариант.

### A4) “Не стрелять если expectedDamage≈0 + muzzle_blocked высокий”

Текущее:
- Бот может стрелять даже при отрицательном скоре и нулевой ожидаемой пользе, что приводит к `noRes=1`.

Предлагается:
- Перед выполнением `execute_fire` (и особенно для hitscan) проверять последний debug:
  - если `chosen.expectedDamage <= 0.05` и `rejected.muzzle_blocked` выше порога (например >= 250), то:
    - отменить fire,
    - форсировать reposition: `plan.action.weaponIndex = -1`, `plan.moveTo = ...` (ближайшая разумная точка / путь / rope стратегия).

## Критерии успеха

- Для матча типа `aivai_1777734550301_hard_hard`: резкое падение `bot_rope_attempt.result:"cooldown"` и рост `attached`, появление решений/выстрелов (decision>0).
- Для матча типа `aivai_1777734373475_hard_hard`: снижение доли `shot_eval.noRes=1` в случаях, где ранее `muzzle_blocked` был высоким; вместо этого должны появиться move/rope попытки.
- В логах `bot_rope_attempt` становится понятно, что именно не получилось: `no_anchor` vs “ray hit” но не attached, и куда был cast.
