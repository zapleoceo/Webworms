# Оружие/экономика/боты: соло-лоадаут, очередь, лимит гранат, UI, фризы кратеров

## Контекст и проблемы
1) В Training (solo) у пользователя фактически доступны только 5 оружий: `bazooka/shotgun/minigun/homing_missile/grenade`.
2) `minigun` (и требуемо `heavy_gun`) должен стрелять очередью фиксированного размера, как в режимах AI/сетевой игры, затем “перезаряжаться” = фактически быть недоступным до следующего хода. В Training оружие не должно “заканчиваться” навсегда.
3) В режимах кроме Training у команды должно быть ограничение: **15 гранат на матч на команду**; боты должны экономить гранаты.
4) При взрыве гранаты наблюдается фриз.
5) Все оружия должны иметь свойство разрушать террейн (возможность включать/выключать кратер — часть характеристик оружия).
6) Оружие в руках у червя нужно уменьшить на 30%.
7) Боты должны оценивать весь новый набор оружия + обновлённую физику гранаты, и использовать гранаты только если попадание гарантировано при ветре ±35.

## Причина “миниган остановился” в Training
Сейчас Training имеет `turnTime = Infinity` и логика автоперехода хода после выстрела (которая есть в остальных режимах) в Training выключена.  
В результате “лимиты на ход” (например `minigun` 25 выстрелов/ход) превращаются в “лимит навсегда”.

## Дизайн изменений

### 1) Единый контроллер расхода (ammo/uses) по команде
- Ввести в `GameState` структуру вида:
  - `teamAmmo: Record<'team1'|'team2', { grenade: number }>`
- Инициализация при `startGame`:
  - `training`: `grenade = Infinity` (или очень большое число)
  - `ai/friend/random/aivai`: `grenade = 15`
- При выстреле гранатой — декремент `teamAmmo[shooter.team].grenade`, если 0 — запрещать выстрел.
- UI должен отображать текущее значение для активной команды.

### 2) Очередь для minigun/heavy_gun (одна очередь на ход)
- Ввести параметр `burstShotsPerTurn` для `minigun` и `heavy_gun` (одно значение для обоих).
- Стрельба:
  - Пока `fire` удержан — создавать 1 “пулю”/“импакт” на тик cooldown, пока `shotsFiredThisTurnByWeaponId[weaponId] < burstShotsPerTurn`.
  - Когда лимит достигнут — оружие больше не стреляет до следующего хода.
- В Training: обеспечить, что после выстрела ход может завершиться (см. пункт 3), чтобы очередь не “заклинивала” навсегда.

### 3) Turn-логика в Training
- Training должен оставаться “без таймера”, но поведение “выстрелил → дождались стабилизации → следующий ход” должно работать как в остальных режимах.
- Цель: чтобы ограничения “на ход” работали одинаково, и Training не ломал экономику/перезарядку.

### 4) “Разрушаемость” террейна как характеристика оружия
- В `Weapon` добавить поле `crater: boolean` (true/false).
- При попадании/взрыве:
  - для projectile/grenade/homing: использовать `weapon.crater`
  - для hitscan/stream: передавать `crater: weapon.crater` в `explodeAt(...)`
- Значение по умолчанию для всех текущих оружий: `true` (требование “всё разрушает”).

### 5) UI: счётчик гранат на карточках 3 юнитов
- В `updateWormSelectionUI` (main.ts) при построении кнопки червя:
  - если `equipmentId === 'grenade'`, добавить небольшой текст в нижней части карточки: `grenades: N`
  - `N` берётся из `presenter.state.teamAmmo[worm.team].grenade` (или аналога).

### 6) Визуальный размер оружия в руках
- В `CanvasRenderer.drawHeldEquipment` умножить базовый размер на `0.7`.

### 7) Боты и “экономная граната”
- В снапшот мира для бота добавить `grenadesRemainingByTeam` и `isTraining` (или mode).
- В `BotAI`:
  - если `grenadesRemainingByTeam[team] <= 0` — не рассматривать гранату как кандидат.
  - если гранаты ограничены — рассматривать гранату только если траектория попадает в цель **для ветра -35/0/+35** (минимум 3 симуляции) и “miss” <= порога.
  - дополнительно в scoring добавить штраф за использование гранаты, чтобы она выбиралась только если это лучший “гарантированный” вариант.
- Обновить симулятор для гранаты в `PhysicsHelper/simulateTrajectory`, чтобы он приближенно соответствовал runtime (учёт SDF-коллизий и скатывания не требуется полностью, но bounce/контакт и отсутствие туннелинга должны совпадать по ключевым случаям).

### 8) Фриз при взрыве гранаты
Сильный кандидат причины: в `CanvasRenderer.drawLandscape` при обработке `newCraters` есть вызовы `getImageData/putImageData` для восстановления пикселей материала `255`, что вызывает синхронный GPU→CPU readback и может фризить кадр.

Фикс:
- Сгенерировать отдельный `alloyCanvas` (слой “несокрушимых пикселей”, материал 255) один раз после генерации карты.
- После применения кратеров в основном `terrainCanvas` просто “подрисовать” соответствующий прямоугольник из `alloyCanvas` обратно через `drawImage` (без `getImageData`).

## Проверка/критерии готовности
- В Training доступны все оружия из `FULL_LOADOUT` и `ninja_rope`.
- `minigun` и `heavy_gun` имеют одинаковый размер очереди и после неё не стреляют до следующего хода; Training не “заклинивает”.
- Во всех режимах кроме Training гранаты ограничены 15 на команду; UI показывает счётчик на карточках, только если выбрана граната.
- Все оружия создают кратер (по `weapon.crater=true`).
- Фриз при взрыве гранаты устранён (нет `getImageData` в hot path кратеров).
- Бот:
  - учитывает ограничения гранат
  - бросает гранату только при гарантированном попадании в диапазоне ветра ±35.
